# Test Writing Style

テストを「通るコード」ではなく、仕様として読める検証に整える。
既存のテスト文化を優先し、読み手が失敗時に契約をすぐ理解できる形にする。

## When to use

- テストを新規追加するとき。
- 既存テストを修正・リファクタリングするとき。
- テストは通るが、名前・構造・関心分離が読みづらいとき。
- 正常系・異常系・境界条件の切り分けをレビューするとき。
- fake / mock / fixture / helper の使い方を既存スタイルへ揃えるとき。

使わない場面:

- テストコマンドを実行するだけのとき。
- CI 失敗の原因調査だけをするとき。
- テストフレームワークや runner の導入・設定だけをするとき。

## Workflow

1. **既存テストを読む**
   - 同じ言語、同じ層、同じ機能周辺のテストを先に確認する。
   - naming、grouping、fixture、helper、assertion library、mock/fake の流儀を既存に寄せる。
   - 新しい helper や抽象化は、既存パターンで表現できないときだけ追加する。

2. **検証する契約を 1 文で言語化する**
   - 「どの条件で、何を保証するか」を書き出す。
   - 実装手段ではなく、外から観測できる結果・副作用・エラー処理を契約にする。
   - その 1 文がテスト名の土台になる。

3. **ケースを分ける**
   - 正常系、異常系、境界条件を別 test / 別 group に分ける。
   - 空入力、未知値、欠損値、権限なし、外部 API 失敗、保存失敗、未対応 platform、二重実行などを「ついで」に混ぜない。
   - 1 つのテストは 1 つの仕様・失敗モード・境界条件を守る。

4. **Arrange / Act / Assert で構造化する**
   - 複雑なテスト、外部依存を含むテスト、UI / E2E 寄りのテストでは `Arrange`、`Act`、`Assert` コメントを置く。
   - Act は原則 1 操作に絞る。
   - 短い純粋関数テストでは、読みやすい場合に限り `Arrange & Act` や `Act & Assert` にまとめてよい。

5. **外部依存を隔離する**
   - 実ネットワーク、実クラウド、実 OS 設定、実時刻に依存しない。
   - 外部境界は mock、fake、in-memory store、local test server、captured output などに置き換える。
   - 時刻、環境変数、一時ファイル、グローバル状態は helper と cleanup で閉じ込める。

6. **副作用の契約を検証する**
   - mock / fake では「呼ばれたこと」だけでなく、渡された値、呼び出し回数、呼ばれてはいけないものが呼ばれていないことを確認する。
   - fake は必要最小限の状態だけを持ち、挙動を決定的にする。

7. **assertion を安全に並べる**
   - 後続検証に必要な前提は fatal assertion で止める。
   - 独立した値比較は non-fatal assertion で複数確認できるようにする。
   - null / nil / length などを確認せずに深いフィールドへアクセスしない。

8. **表で読めるロジックはケース表にする**
   - 純粋関数、変換ロジック、入力と期待値の組み合わせで表せる処理は table-driven / parameterized test を検討する。
   - 各ケースには条件が分かる名前を付ける。
   - 複雑な副作用シナリオは、無理に表にせず個別 Arrange を書く。

9. **設定ファイルも契約として扱う**
   - CI workflow、Terraform、dependency policy、生成物の形式など、壊れると困る設定はテストで固定する。
   - 実行せずに検証できる契約は、静的なファイル読み込みや plan 検証で守る。

## Checklist

テスト追加・修正後に確認する。

- テスト名は「条件 + 期待結果」を仕様文として読めるか。
- 既存テストの naming / grouping / helper / assertion style に揃っているか。
- 1 テスト 1 関心になっているか。
- 正常系・異常系・境界条件が混ざっていないか。
- Arrange / Act / Assert が読み取れるか。
- Act が 1 操作に寄っているか。
- 外部依存が実物へ接続していないか。
- mock / fake で引数、回数、呼ばれないことを必要に応じて検証しているか。
- 前提確認と値確認が分かれているか。
- テスト失敗時に、どの契約が壊れたか分かるか。

## Pitfalls

- テストが通っただけで、既存テストの書き方に揃えない。
- 1 つのテストで正常系、異常系、副作用、境界条件をまとめて確認する。
- 実装詳細をテスト名にして、外から見える仕様が分からない。
- mock を依存の穴埋めだけに使い、副作用の契約を検証しない。
- 実ネットワーク、実時刻、実 OS 状態、グローバル状態に依存して flaky にする。
- helper を先に抽象化しすぎて、個別ケースの Arrange が読めなくなる。
