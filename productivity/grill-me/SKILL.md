計画・設計・方針・タスク分解を、実行前に失敗しにくい形へ追い込む。
ユーザーが求めているのは同意ではなく、抜け・曖昧さ・未検証の前提を露出させることだと扱う。

## Trigger contract

この skill を使うのは、ユーザーが計画・設計・方針・タスク分解について、実行前に次のような判断を求めたとき。

- このまま進めてよいか。
- 方針として妥当か。
- 考慮漏れや危ない前提がないか。
- 先に確認すべきことがあるか。
- 実装へ移る前に壁打ちしたい。

次の場合は使わない。

- 既に書かれたコード差分や PR の correctness / security / maintainability レビュー。
- 実装済み成果物の通常レビュー、文章の推敲、学習用の説明。
- ユーザーが計画作成そのものを求めており、批判的検証を求めていない場合。

## Workflow

1. 対象 artifact を特定する。
   - 計画、設計メモ、issue、会話中の案など、何を詰めるのかを明示する。
   - 対象が曖昧なら、最初の質問で「何を grill するか」を確認する。

2. コードベースや既存文書を調べれば答えられる前提は、ユーザーに聞かずに自分で調べる。
   - 調査した事実と、まだ未確認の仮定を分ける。
   - 調査なしに決めつけて質問しない。

3. まず最も失敗確率を上げる不確実性を 1 つ選ぶ。
   - 例: 要件の境界、成功条件、非目標、既存制約、移行手順、ロールバック、観測性、データ整合性、セキュリティ、ユーザー影響、運用負荷。
   - 好みや表現の細部より、後戻りコストが高い前提を優先する。

4. 質問は一度に 1 つだけ行う。
   - 形式は「対象: ...」「詰めどころ: ...」「質問: ...」「推奨/暫定判断: ...」を基本にする。
   - 各質問には、なぜその質問が詰めどころなのかを 1 文で添える。
   - 可能なら推奨回答または暫定判断も添える。
   - ユーザーの返答を待ってから次へ進む。複数の質問を同時に投げない。

5. ユーザーの返答ごとに、意思決定ツリーを更新する。
   - 解消した前提、残った前提、新しく生じたリスクを短く整理する。
   - 次に聞く質問は、その時点で最も影響が大きい 1 問に絞る。

6. 共有理解に到達したら、最終的な実行条件を短くまとめる。
   - go / no-go 条件。
   - 残リスクと受容理由。
   - 次に実装・調査・文書化すべき具体アクション。
